# PRD: Backend Processing Pipeline

## Design principle

Default to deterministic/classical algorithms. Escalate to an LLM only where
the task is reasoning over ambiguous language/context, do not default to audio perception via LLM. Every LLM call in this pipeline should be justifiable as "an algorithm cannot
make this judgment call," not "an LLM would also work."

---

## 1. Recognition

Handled entirely client-side by site adapters (see `prd-extension-client.md`).
The backend receives already-identified metadata; no recognition logic lives
server-side.

---

## 2. Copyright check — tiered cascade, blocking in production -- fingerprinting is in V0 but check vs. copywriting is optional for V0.

No admin toggle for notify-only in prod. A blocked capture never reaches the
tagging stage or the database.

**Tier 1 — Chromaprint / AcoustID fingerprint match**
Deterministic, near-zero cost. Catches exact/near-exact rips. Query against
the open AcoustID database first; coverage gaps expected for long-tail or very
recent commercial tracks.

**Tier 2 — Landmark-hash matching (Panako or equivalent)**
Deterministic. Specifically targets pitch-shifted/time-stretched audio, which
plain fingerprinting misses and which producers commonly produce.

**Tier 3 — Commercial fingerprinting API fallback**
Paid, called only on Tier 1/2 misses (not per-capture) to cover catalog gaps
in the open databases. Vendor TBD — budget-dependent, see open decisions in
`prd-overview.md`.

**Tier 4 — LLM reasoning agent (only remaining ambiguous cases)**
Inputs: uploader type signal, upload date vs. estimated original release date,
source platform, title/description text, and the fact that Tiers 1-3 found no
match. Outputs: block/allow decision + a short logged rationale (private audit
log only, never shown to the end user).

This tier should encode general norms (how commercial rips typically get
mislabeled, what "official audio" / "topic channel" conventions imply,
how radio-archive vs. platform-native content differs) as a rubric in the
system prompt — there is no clean labeled dataset for this, so the rubric
itself is a first-class artifact that should live in version control and get
refined via the aggregate-pattern feedback loop (see `prd-hitl-review.md`).

**Cost control:** the whole point of the cascade order is that the large
majority of captures resolve at Tier 1 or 2, for effectively free. Tier 4
volume should be monitored — a spike suggests Tiers 1-3 need tuning, not that
Tier 4 needs to get smarter.

## 3. Tagging — audio ML models first, LLM strictly as judge (never as tagger)

**Origin/rationale (see `prd-overview.md`, "Status quo & why this exists"):**
an earlier version of this pipeline tagged directly via an LLM call per
sample. Deterministic properties (BPM, Key) were stable that way, but
subjective properties (Genre, Mood) stayed probabilistic regardless of
prompting — LLMs are inconsistent audio-perception tools, full stop. That's
why every tag below is generated by a local model or classical algorithm, and
the LLM's only role is judging confidence and deciding whether a tag needs a
producer's eyes, never generating the tag itself.

**Algorithmic / audio ML (no LLM, runs on every cleared capture):**

*BPM/Key (deterministic):*
- BPM: `librosa.beat.beat_track` (ISC) as primary, cross-checked against
  `aubio`'s tempo tracker (GPLv3 — **copyleft, flagging plainly per
  CLAUDE.md constraint #4's "anything else copyleft" clause**, not just
  AGPL). Running two independent algorithms isn't redundancy for its own
  sake — it's how `bpm_confidence` gets computed concretely: agreement
  between librosa and aubio → high confidence; disagreement → low
  confidence, which is what actually drives the LLM-as-judge threshold
  below. Previously this field had no defined mechanism for where its
  confidence number came from; this is that mechanism.
- Key: `essentia` KeyExtractor (AGPL — already flagged; kept narrowly
  scoped to Key specifically, where it's genuinely better than a DIY
  chroma-correlation approach in librosa, per the existing licensing note). A 2nd or 3rd key algorithm can be chosen from the LLM and added here.

*Considered and deliberately excluded — `madmom` (BSD source /
**CC BY-NC-SA 4.0 pretrained models**):* its beat/downbeat tracking is
state-of-the-art, but that quality comes from the pretrained neural models,
which are **non-commercial only**. Since this product may not stay a
strictly non-commercial research project forever, adopting madmom now would
plant a license landmine for later. Not included; revisit only if the
project commits to staying non-commercial or CPJKU offers a commercial
license.

*One-shot vs. loop:* heuristic (duration + onset density + tempo-lock
check), not ML — unchanged.

*Category/instrument (vocal/drum/pad/fx/kick/snare/etc.):* PANNs (MIT) or
YAMNet (Apache 2.0), pretrained on AudioSet, run locally — unchanged, both
confirmed permissively licensed. **Multi-label, not single-value:** PANNs/
YAMNet score every AudioSet class independently (sigmoid outputs, not a
single softmax pick), so `categories` should store every class that crosses a
tagging-display threshold, not just the top-1 label — a captured loop with
both a drum layer and a melodic layer should show as both, not be forced
into one. This is also the input the stem-extraction trigger in section 4
depends on — see there for how the raw multi-label output gets used.

*Genre/Mood (probabilistic — this is the split the origin-story rationale
above is actually about):*
- **Genre:** `musicnn` (Jordi Pons; ISC), pretrained on MagnaTagATune/Million
  Song Dataset tag vocabularies. Cross-checked against PANNs/YAMNet's own
  genre-adjacent AudioSet labels for the same agreement-based confidence
  mechanism as BPM above.
- **Mood:** `CLAP` (LAION-AI; **CC0 — public domain**, the most permissive
  option surfaced in this research). Rather than a fixed mood-class list,
  CLAP does contrastive text-audio embedding matching — score the capture's
  audio embedding against a candidate set of mood-phrase embeddings ("dark
  ambient", "moody trap", etc.) and take the top match + margin as the
  confidence signal. This is a deliberate methodology choice, not just a
  license pick: mood is exactly the kind of open-ended, personal-language
  property a fixed classifier's taxonomy can't fully capture, and CLAP's
  open-vocabulary matching fits that better than forcing mood into a fixed
  56-class list.

*Considered and deliberately excluded:*
- **Essentia's own pretrained Genre-Discogs400/Mood-Jamendo models** —
  **CC BY-NC-ND 4.0** (non-commercial *and* no-derivatives, stricter than
  madmom's terms). Same non-commercial problem as madmom, stacked on top of
  the AGPL code license already flagged for essentia's Key extractor.
- **OMAR-RQ** (MTG, 2025 SOTA self-supervised music model) — **CC-BY-NC-SA
  4.0**, same non-commercial issue. Worth revisiting if a commercially
  licensed variant appears; not adoptable as-is.
- **MOSS-Music** (OpenMOSS, 8B-parameter multimodal music-understanding
  model) — excluded on architectural grounds, not just license: it's an
  LLM-based approach to tagging, which is precisely what the origin-story
  rationale above says didn't work (inconsistent on subjective properties)
  and precisely what constraint #3 (near-zero runtime cost, LLM only for
  genuine language/context reasoning) argues against. Running an 8B model
  per capture to do what a small pretrained classifier already does cheaper
  and more consistently would undo the whole point of this redesign. -- Update on 7/14/2026 - i am OK to use an LLM if there is no current cost to the user. MOSS-MUSIC is open source and may be low-runtime cost. this is a useful model upon review.

Genre and Mood are new tag categories as of this pipeline revision — also
reflected in `prd-database-schema.md`'s `Tags` entity. Every field above
gets its own confidence score, derived from cross-model agreement where two
models are run (BPM, Genre) or from match-margin where one embedding-based
model is used (Mood) — not an arbitrary number.

**Licensing note:** essentia's default build is AGPL; aubio is GPLv3 —
**both flagged explicitly here**, not just essentia, per CLAUDE.md
constraint #4's "anything else copyleft" clause. `librosa`, `musicnn`, and
`CLAP` are the permissively-licensed core (ISC/ISC/CC0); PANNs/YAMNet
(MIT/Apache-2.0) round out the permissive set. madmom, Essentia's own
tagging models, and OMAR-RQ are excluded above specifically for their
non-commercial license terms — re-evaluate if the project's commercial
status changes.

**LLM-as-judge — confidence gate at 0.7, runs across all tag fields above:**
Not a tagger. Takes each field's audio-ML-generated value + confidence score
(not raw audio) and decides: confidence ≥ 0.7 → auto-finalize the tag as-is;
confidence < 0.7 → route to the producer for review. This generalizes what
was previously scoped as BPM/Key-only disambiguation — it now runs over
Genre and Mood too, since those are exactly the probabilistic properties the
rationale above describes. When it does route to a producer, the job is
still to frame a clear yes/no or pick-one question, not to re-run its own
audio judgment (see `prd-hitl-review.md` touchpoint 2).

**LLM naming/enrichment agent — runs on every tagged sample:**
Takes the structured tags (e.g. "kick, 128bpm, Cmin, one-shot") and proposes a
human-searchable name (e.g. "deep sub kick, minor, trap-adjacent"). This is a
language task an LLM is well-suited to; the structured tags remain the source
of truth for filtering/search, the generated name is a display/search-alias
layer on top. Producer edits to this name are the input to HITL touchpoint 4.

**Tag-preference feedback loop — resolved:** every tag's audio-ML value, the
LLM-judge's confidence/finalize-or-route decision, and (when triggered) the
producer's final answer get logged as a tuple — same pattern as touchpoint
4's naming tuple, see `prd-hitl-review.md` touchpoint 6. This settles the
previously-open question about what "trains the system on tag-preference"
means: **it cannot mean retraining the audio ML models** — librosa, aubio,
essentia, PANNs/YAMNet, musicnn, and CLAP are all static pretrained
artifacts here, not owned/trainable models. What actually improves over
time is the LLM-as-judge's own prompt/instructions — the corrections feed
back as context for arbitration, not as gradient updates to a tagger.
Concretely: the aggregate pattern agent (section 5) periodically mines
tag-correction tuples for systemic patterns (e.g. "the judge keeps
auto-finalizing 'moody' when producers consistently correct to 'dark
ambient'") and proposes a judge-prompt revision — the same
propose-to-`architecture-notes.md`-for-a-dev-agent-to-apply mechanism
already used for naming feedback, not a new training pipeline.

**Retrieval-augmented judging — the concrete near-term direction, one step
past static prompt edits:** rather than only judging the current sample's
`Tags.raw_model_outputs` in isolation, the judge's call should also be given
retrieved context — per-model historical accuracy (mined from how often
each model's candidate in `raw_model_outputs` has matched
`producer_final_tag` in past `tag_correction` events) and relevant past
`producer_comment` text (the actual reasoning behind past corrections, not
just the corrected value). This is a distinct third mechanism from the two
already discussed, worth naming precisely so they don't get conflated:

| Mechanism | What changes | How | Cost |
|---|---|---|---|
| Static prompt refinement (built above) | The judge's written instructions | A dev agent periodically edits the prompt by hand | One-time edit, no runtime cost |
| **Retrieval-augmented judging (this)** | What context the judge sees per-call | Retrieve relevant historical stats/comments, inject as context, same prompt | Small retrieval cost per call, no training |
| Trained arbitrator (discussed, not planned) | The decision logic itself | Fit a small model's weights to `raw_model_outputs` → `producer_final_tag` via gradient descent | Training pipeline + ongoing retraining |

Retrieval-augmented judging needs no new infrastructure beyond what's
already in the schema (`Tags.raw_model_outputs`, `HITLEvent.tag_correction`)
plus a retrieval step before the judge call — it's still squarely "the
judge's context improves," not training, so it doesn't reopen the
"can't retrain static models" resolution above. This is the next concrete
build step for the judge, ahead of the trained-arbitrator option.

**Longer-term direction — producer-aligned recommendations (aspiration,
not yet spec'd):** maintainer's stated goal is for tag recommendations to
eventually align with either the specific producer (personalization) or a
broader producer-community consensus, not just be generically "more
accurate." This is a real scope expansion worth being explicit about rather
than quietly folding in:
- **Personalization requires producer identity to exist at all.** Nothing
  in the schema currently attributes a capture or a correction to a
  specific producer — see `producer_id` added to `Sample` and `HITLEvent`
  in `prd-database-schema.md` specifically so today's single-local-user
  data doesn't have to be thrown away once multi-producer support exists.
- **Resolved:** "the larger human producer community" means aggregating
  preference data across this product's own multi-user base once it has
  one at production scale — not blending in external community-tagged
  datasets (MagnaTagATune/Discogs/AudioSet-style vocabularies stay scoped
  to what they already do, training the base models, per section 3 above).
  This means no new schema is needed beyond `producer_id`
  (`prd-database-schema.md`) — "individual" alignment is `tag_correction`
  events filtered to one `producer_id`; "community" alignment is the same
  events aggregated across all of them. One data source, two query shapes,
  not two separate pipelines. At current scale (single local producer) this
  entire axis is moot — nothing to build until there's more than one
  producer generating corrections; the point of `producer_id` existing now
  is purely so today's data isn't discarded when that day comes.
- **Open question, not resolved here:** how individual-producer preference
  and community consensus get reconciled when they conflict (a specific
  producer always calls something "riser" when the community calls it
  "sweep" — which wins, and under what conditions), and how a brand-new
  producer with no correction history gets a reasonable starting point
  (cold start) before any personalization signal exists.
- This direction likely implicates the trained-arbitrator path above more
  than retrieval-augmented judging does — per-producer weighting is closer
  to "fit a model to this producer's correction history" than "retrieve
  relevant context for a general judge" — but that's a design call for
  whoever scopes this once there's enough multi-producer data to make it
  real, not assumed here.

## 4. Stem extraction — multi-algorithm, DSP-scored (no LLM judging audio)

**Auto-detection flag vs. manual trigger — these are now two separate
things, not one gate.** Originally this section read "runs only when
tagging flags the capture as layered/multi-instrument music," which
conflated "the system suggests this" with "this is the only way to invoke
it." Per maintainer decision: the "Extract Stems" button is **always
available** to the producer regardless of the flag — extraction is a user-
agency action, not something purely gated by the algorithm. The flag's job
is narrower: decide whether to proactively *suggest* extraction (see
`prd-hitl-review.md` touchpoint 7), not whether to allow it.

**Multi-instrument detection flag — concrete trigger condition:**
- Uses the multi-label `categories` output from section 3 (PANNs/YAMNet),
  collapsed into the same four stem-relevant buckets `Stem.stem_type` and
  HTDemucs already use: vocals / drums / bass / other. Collapsing to these
  four (rather than counting raw AudioSet classes) matters — AudioSet's
  ontology is hierarchical and overlapping (e.g. "Drum kit" and
  "Percussion" can both score high for one single layer), so counting raw
  label hits would false-positive constantly.
- Flag fires when **≥2 of those four buckets** cross a **separate**
  confidence threshold — recommended starting point **0.5**, deliberately
  lower and deliberately not reusing the 0.7 LLM-as-judge tagging
  threshold, since this answers a different question ("is more than one
  layer present") than tagging confidence does ("is this classification
  reliable"). **Maintainer has flagged both numbers as starting points to
  tune once the product is working, not final** — treat 0.5/≥2-buckets as
  a placeholder methodology, not a locked spec.
- Cost note: because extraction is now manually invokable regardless of the
  flag, the automatic cost gate this feature used to have is gone —
  producers can trigger the most expensive step in this pipeline on any
  sample, flagged or not. Not a reason to block the UX decision, but
  extraction volume on *non-flagged* samples should be monitored as a cost
  variable the same way Tier 4 copyright volume already is (section 2),
  since unlike the flagged case it isn't self-limiting.

- Run candidate separations from more than one model where budget allows:
  HTDemucs (general 4-stem, current open-source SOTA) and Mel-Roformer or
  MDX-Net (vocal-isolation specialists) — pick per use case rather than always
  running all three, to manage compute cost.
- Score outputs with an objective DSP metric (spectral leakage / SDR-proxy
  estimating bleed between separated stems), not an LLM listening pass.
- Periodic (not per-file) multimodal-LLM spot-checks on a small sample of
  outputs to validate the DSP scorer stays aligned with human judgment over
  time — a calibration task, not a runtime dependency.
- Stems are stored as children of the raw-file record in the database
  (see `prd-database-schema.md`).

## 5. Aggregate pattern / error-detection agent

Not part of the per-capture hot path. A periodic job (e.g. daily) that reads
STATUS.md, the naming-feedback logs, and the Tier 4 copyright audit log, and
writes findings to `architecture-notes.md` for dev agents to act on next
session. See `status-log-protocol.md` and `prd-hitl-review.md`.
